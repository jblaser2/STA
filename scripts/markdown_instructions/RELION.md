# RELION — Subtomogram Classification (3D input) Runbook

> How to classify **already-reconstructed, pre-aligned 3D subtomograms** with RELION on this
> machine, and why the version matters. Written for the STA benchmark (672 T4P particles, 80³,
> 13.328 Å/px). Mirrors the depth of `protomo/research.md`.

## 1. Which RELION version does 3D-subtomogram classification?

**Short answer:** the *documented, GUI-driven* workflow for classifying **already-reconstructed 3D
subtomograms** is the **RELION 3.0/3.1** "classic" subtomogram-averaging pipeline. RELION **4.0**
and **5.0** replaced it with **pseudo-subtomograms**.

| | RELION 3.0 / 3.1 (classic) | RELION 4.0 / 5.0 (pseudo-subtomo) |
|---|---|---|
| Particle model | a real, reconstructed **3D subvolume** (`.mrc`) + a **3D CTF/wedge model** (`rlnCtfImage`) | a **pseudo-subtomogram** built on-the-fly from the **2D tilt-series images** |
| Optimisation target | likelihood over **3D** data | likelihood approximating the **2D** experimental images |
| Requires tilt series? | **No** — works from extracted subvolumes alone | **Yes** — needs tilt series + tomogram geometry + per-tilt CTF |
| Tool to prep CTF | `relion_prepare_subtomograms.py` → `do_all_reconstruct_ctfs.sh` | `relion_tomo_*` pipeline (import tomograms, make pseudo-subtomos) |
| 3D classification | `relion_refine --K N` on the 3D star | `relion_refine --K N` on the pseudo-subtomo set |

For our dataset we have **only 672 pre-aligned 80³ `.mrc` volumes — no tilt series, no tomograms,
no per-tilt CTF**. So the modern pseudo-subtomogram pipeline is **not directly applicable**; the
classic 3D-subtomo path is the right fit. The user's premise ("only RELION 3.1–4.0 did STA
classification on 3D data") is therefore essentially correct.

### The key nuance — RELION 5 still *contains* the classic 3D path

The classic 3D-subtomo code was **not removed** from the `relion_refine` binary in RELION 5; it is
just no longer exposed through the GUI tomo pipeline. Verified locally on the
`5.0.1-commit-cad71b` build:

- `relion_refine --help` shows a **"Subtomogram averaging"** section:
  `--normalised_subtomo`, `--skip_subtomo_multi`, `--ctf3d_not_squared`, `--subtomo_multi_thr`.
- The 3D-CTF field is defined: `rlnCtfImage` ("Name of an image with all CTF values")
  — `src/metadata_label.h:886`.
- With 3D data and `--ctf`, `relion_refine` **requires** `rlnCtfImage` per particle, else it errors:
  `src/ml_optimiser.cpp:10343` (`"cannot find rlnCtfImage for 3D CTF correction!"`).
- 3D mode is triggered by `rlnImageDimensionality 3` in the **optics** block:
  `src/exp_model.cpp:888`.
- The 3D-CTF cube is read as a centered full cube (`XSIZE==YSIZE==ZSIZE`):
  `src/ml_optimiser.cpp:get3DCTFAndMulti` (~L10557). A plain 80³ cube **bypasses** the
  subtomo-multiplicity branch (`ZSIZE==YSIZE*2`), which is what we want with `--skip_subtomo_multi`.

**Conclusion:** we can run classic 3D Class3D directly on the installed **RELION 5** from the
command line — no separate 3.1 build required. (A dedicated 3.1 build is documented in §7 as a
fallback for benchmark fidelity.)

## 3. Input data & required STAR format

- 672 aligned subtomos: `subtomos_mrc/aligned_tomXXX_PYYYY.mrc` — 80³, 13.328 Å/px, float32.
- Pre-aligned & centered: angles/origins = 0 (see `outputs/motl.txt`).

RELION ≥3.1 uses a **two-block** star (`data_optics` + `data_particles`). The optics block sets the
3D flag and pixel size; the particle block lists each subvolume and its (shared) 3D CTF model:

```
data_optics
loop_
_rlnOpticsGroup #1
_rlnOpticsGroupName #2
_rlnImagePixelSize #3      # 13.328
_rlnImageSize #4           # 80
_rlnImageDimensionality #5 # 3  <- triggers 3D mode
_rlnVoltage #6             # 300  (placeholder)
_rlnSphericalAberration #7 # 2.7  (placeholder)
_rlnAmplitudeContrast #8   # 0.1  (placeholder)
1 opticsGroup1 13.3280 80 3 300.0 2.7 0.1

data_particles
loop_
_rlnImageName #1
_rlnCtfImage #2            # shared 3D wedge or uniform model
_rlnOpticsGroup #3
_rlnAngleRot #4 _rlnAngleTilt #5 _rlnAnglePsi #6        # all 0 (pre-aligned)
_rlnOriginXAngst #7 _rlnOriginYAngst #8 _rlnOriginZAngst #9   # all 0 (centered)
subtomos_mrc/aligned_tom100_P0001.mrc outputs/relion/ctf/wedge_ctf.mrc 1 0 0 0 0 0 0
...
```

Generated by **`scripts/data_prep/build_relion_star.py`** (autodetects box/pixel via the same logic
as `generate_motl.py`), producing `particles_wedge.star` and `particles_uniform.star`.

## 4. The 3D CTF / missing-wedge model

No tilt series ⇒ no per-particle CTF. We supply **one shared 3D model** reused for all 672
particles (they share the same tomogram orientation), in two variants for comparison:

- `wedge_ctf.mrc` — single-axis missing-wedge mask (tilt axis = Y, ±60°): 1.0 in the measured
  region, 0.0 in the missing wedge (≈71% measured). Down-weights missing-wedge directions during
  classification.
- `uniform_ctf.mrc` — all-ones cube: no CTF/wedge weighting (controlled baseline).

Built by **`scripts/data_prep/make_wedge_ctf.py`** as centered full 80³ cubes (the layout
`get3DCTFAndMulti` expects).

## 5. Initial reference

Because the particles are already aligned (angles=0), a plain voxel-wise mean is a valid unbiased
seed. **`scripts/data_prep/make_initial_ref.py`** averages all 672 subtomos →
`outputs/relion/initial_ref.mrc`; RELION low-pass filters it at startup via `--ini_high 60`.

## 6. Running Class3D (the validated command)

Driver: **`scripts/data_prep/run_relion_class3d.sh`** loops k ∈ {2,3,4} × CTF ∈ {wedge, uniform}.
Run **from the STA project root**. Representative command:

```bash
/home/jblaser2/relion-install/bin/relion_refine \
  --i outputs/relion/particles_wedge.star \
  --ref outputs/relion/initial_ref.mrc \
  --o outputs/relion/Class3D/k2_wedge/run \
  --K 2 --iter 25 --tau2_fudge 4 --ini_high 60 \
  --particle_diameter 960 --skip_align \
  --sym C1 --ctf --skip_subtomo_multi --zero_mask --pad 2 --norm --scale \
  --dont_combine_weights_via_disc --j 8 --gpu ""
```

**Flags that matter for 3D-subtomo input (learned the hard way):**

| Flag | Why |
|---|---|
| `--ctf` + `rlnCtfImage` | 3D CTF correction; **mandatory** field for 3D+ctf or `relion_refine` aborts |
| `--skip_subtomo_multi` | our CTF cubes are plain 80³ (no multiplicity half) — skip the pseudo-subtomo multiplicity correction |
| `--zero_mask` | **required on GPU+3D**: default noise-masking is "hard-coded 2D" and aborts (`src/acc/acc_ml_optimiser_impl.h:365`). Mask solvent with zeros instead |
| `--skip_align` | pure classification: particles are already aligned, so keep poses fixed (also matches how PyTom/Protomo were run, and is much faster) |
| `--ini_high 60` | low-pass the initial reference to 60 Å to avoid reference bias |
| `--particle_diameter 960` | ≈ 72 px × 13.328 Å, just inside the 80³ box |
| `--gpu ""` | use the RTX 5080 |

Drop `--skip_align` (and add `--healpix_order 1 --offset_range 8 --offset_step 2`) if you want
RELION to *also* do a local pose search — slower, and it re-aligns the particles.

Outputs per run: `run_it025_class00N.mrc` (class maps), `run_it025_model.star` (occupancies +
resolution), `run_it025_data.star` (per-particle `rlnClassNumber`). All under
`outputs/relion/Class3D/` (large — gitignored).

## 7. (Fallback) canonical RELION 3.1 build

If the benchmark needs the *published* classic pipeline rather than RELION 5's retained path:
build RELION 3.1 and use `relion_prepare_subtomograms.py` (runs CTFFIND on the tilt stacks and
writes `do_all_reconstruct_ctfs.sh` to reconstruct per-particle 3D CTF models) → `particles_subtomo.star`
→ GUI 3D-classification. **Caveat on RHEL10 / CUDA 13.2:** RELION 3.1 predates modern CUDA; expect
CMake/CUDA-compat pain — likely a CPU-only build or a legacy CUDA toolkit. Only attempt if reviewers
require it; the wedge models above are a stand-in for the per-particle CTFs this dataset can't provide.

## 8. Analysis

**`scripts/analysis/relion_class_report.py`** parses each run's final `model.star`/`data.star` for
class occupancy + resolution, computes inter-class cross-correlation from the class maps, and renders
central XY/XZ/YZ slices per class → `outputs/relion/results/` (PNGs + `RESULTS.md`).

## 9. Results

Run 2026-05-29 on the 672 T4P particles, RELION 5.0.1, `--skip_align`, 25 iters. Full table +
slice figures in `outputs/relion/results/`.

| Run | k | CTF | Class sizes (occupancy) | Inter-class CC |
|---|---|---|---|---|
| k2_wedge   | 2 | wedge   | 656 (98%), 16 (2%) | 0.971 |
| k2_uniform | 2 | uniform | 637 (95%), 35 (5%) | 0.981 |
| k3_wedge   | 3 | wedge   | 574 (85%), 50 (7%), 48 (7%) | 0.990–0.996 |
| k3_uniform | 3 | uniform | 608 (90%), 39 (6%), 25 (4%) | 0.983–0.993 |
| k4_wedge   | 4 | wedge   | 573 (85%), 42, 29, 28 | 0.987–0.997 |
| k4_uniform | 4 | uniform | 598 (89%), 32, 25, 17 | 0.979–0.992 |

**Interpretation.** Across every k and both CTF treatments, RELION produces **one dominant class
(85–98%) plus a few small outlier groups**, and the class averages are **near-identical**
(inter-class cross-correlation **0.97–0.997**). The class maps are genuine pilus density (periodic
filament subunits visible in XY; missing-wedge smearing in XZ/YZ) — i.e. the classifier *is*
working; it simply finds no balanced discrete split. This **corroborates PyTom** (k=2/k=3 averages
identical) and **Protomo** (inter-class CC = 0.921): three independent packages converge on
"no strong discrete heterogeneity" for this T4P set.

**Wedge vs uniform.** The missing-wedge mask slightly changes *which* particles get peeled into the
small classes but does not create separation — the wedge is **not** the thing masking a real signal.
This makes the convergent null result more credible and points the open scientific question
("is T4P discrete at all?") squarely at Stefano + the synthetic ground-truth datasets.

**Caveats.** Imbalanced tiny classes (a handful of particles) are RELION's classic behaviour when
the dominant signal is one state plus noise/outliers; they are not necessarily distinct
conformers. Resolution estimates per class are low (small particle counts, no real CTF). The
shared synthetic wedge is a stand-in for the per-particle CTFs this tilt-series-free dataset
cannot provide.
